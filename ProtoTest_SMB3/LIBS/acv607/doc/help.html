ASN1C Compiler Wizard User's Guide

July 2006



Introduction

This document provides tutorial information for the ASN1C Compiler Wizard, a graphical application that acts as a front end to Objective Systems' ASN1C command-line compiler. Documentation related to the ASN1C compiler and runtime library may be found on-line at http://www.obj-sys.com/docs/documents.shtml. This document is not intended to be a comprehensive reference for the command-line tool. Consult the documentation when questions arise about compiler use.

The Compiler Wizard is navigated by means of Next and Back buttons. Check boxes, text-entry forms, and radio buttons are used to select compiler options.

The Wizard contains several context-sensitive options, so some items may be inactive or invisible when others are active. For example, it is impossible to specify both BER and DER encoding rules simultaneously. When the target language is C#, function types that may be generated for C or C++ do not appear at all.

The ASN1C Project Wizard




Users have the option of creating and loading project files during each execution of the Wizard. A project saves all selected compilation options, allowing users to store commonly-used options for later retrieval. To create a new project, click Create a new project. To load a previously saved project, click Open an existing project.

When closing, the Wizard will offer to save the current project, if modified, or to create a new one.

Select Files and Directories




XSD and ASN.1 Files

To add ASN.1 or XSD specifications to be compiled, click the Browse button next to the pane labeled ASN.1 / XSD Files and select files from the browse window that appears. To remove a file, highlight it in the list and click Remove.

Note ASN1C will compile both ASN.1 and XSD files, but it will not compile both at the same time. When attempting to add files, the Wizard will prevent users from adding files of both types. In order to add an XSD file, all ASN.1 files must be removed from the list, and vice versa.

Import Directories

To specify directories for the compiler to search for imported specifications (i.e., those provided by IMPORT declarations in ASN.1 files), click the Browse button next to the Include / Import Directories pane. To remove a directory, highlight it in the list and click Remove.

Configuration Files

Configuration files are added the same way. A configuration file contains directives that tell the compiler how to treat production rules it encounters in an ASN.1 specification. Configuration files have a .cfg or .xml extension. The compiler manual describes the format of these files in detail.

Output Directory

By default, the compiler generates code in the current working directory. Specifying an output directory will cause the compiler to write all of its files to a directory of the user's choosing. The output directory is selected in the same manner as the import directories.

Generate Listing

By default, ASN1C does not display the contents of the specification it compiles. Checking Generate Listing will cause it to echo the specification to the screen.

Output Warning Messages

The same is true of compilation warnings. Checking Output Warning Messages will cause the compiler to produce warning messages if any are appropriate.

Code Generation Options




Input File Type

ASN1C can parse ASN.1 specifications based on both the X.680 standard (post-1997) and the X.208 standard (pre-1997, with ROSE or SNMP macros). When compiling a legacy specification, select the Legacy ASN.1 radio button.

ASN1C also recognizes XML schema definitions (XSD files). When one or more XSD files are specified in the Select Files and Directories window, the XML Schema radio button will be active and the ASN.1 radio buttons will be inactive (grayed out).

Application Language Type

The target language may be C, C++, C#, or Java. In order to perform a simple syntax check without generating any source code, click None.

Additional Translations

Depending on which file type is selected in the Select Files and Directories window, options are also available to translate between ASN.1 specifications and XML Schema Definitions. If the input is an ASN.1 specification, the check box marked Generate equivalent XML schema (XSD) file will be active and the check box marked Generate ASN.1 file based on X.694 (XSD input only) will be inactive. The disposition of the check boxes is reversed when the input is an XSD file.

Dependencies

ASN1C can also attempt to compile all specifications that are imported by the given specification. If different compilation options are required, a configuration may be created that can fine-tune some of the controls.

C/C++ Code Generation Options




Compiler options for the C and C++ target languages are the same.

Select Encoding Rules

ASN1C can generate code compliant with several ASN.1 standards as well as the W3C XML standard:

Encoding Rules

Description

BER

Basic Encoding Rules, defined in ASN.1 standard X.690.

DER

Distinguished Encoding Rules, defined in ASN.1 Standard X.690.

CER

Canonical Encoding Rules, defined in ASN.1 Standard X.690.

PER

Packed Encoding Rules, defined in ASN.1 Standard X.691.

XER

XML Encoding Rules, defined in ASN.1 Standard X.693.

XML

XML Encoding rules, as specified by the World Wide Web Consortium.



Note that only one of BER, CER, and DER may be selected at a time. XML and XER are also mutually exclusive options.

Generated Function Types

Several different functions may be generated for each ASN.1 or XML production. Generating these functions may result in a substantially larger code base, so care should be exercised when code size is a concern. By default, only Encode and Decode functions are selected. In cases where only one or the other is required, a smaller code base may be generated by unchecking the indicated boxes.

Initialization

Check Initialization to generate functions that will initialize each ASN.1 production. This produces code that more efficiently initializes variables for the generated types.

Memory Free

Check Memory Free to generate memory free functions for all generated types. Normally ASN1C handles memory deallocation through a custom memory free function, rtMemFree, that frees all of the memory used by a specific context. By selecting this option, it is possible to achieve finer control over memory deallocation.

Copy Functions

Check Copy to instruct the compiler to create functions that perform a deep copy. This deep copy is performed by ASN1C memory management functions.

Compare Functions

Check Compare to generate functions that will permit generated type variables to be compared to each other. These are debug functions.

Test Functions

Check Test to generate test functions that populate generated types with random data. Use the generated encode functions to encode test constructs. Test functions may also serve as a template for how to populate generated types.

Test functions are outputted to <Module_Name>Test.c, where <Module_Name> is the name of the ASN.1 module.

Stream Functions

Check Stream to generate stream-based encode/decode functions instead of the typical memory-buffer based. These permit users to encode directly to or decode directly from a source or sink such as a file or socket. Stream-based encode and decode routines cannot be combined with memory-buffer-based routines in any way.

The functions will be written to <Module_Name>Print.c where <Module_Name> is the name of the module from the ASN.1 source file.

Named Bit Macros

Check Named Bit Macros to generate functions that generate macros to set, clear, or test the named bit in a bit string structure. These macros offer better performance then using the run-time functions because all calculations of mask and index values are done at compile time. However, this represents a substantial increase in generated code size.

Print Functions

The compiler supports three different printing functions: Print to stdout, Print to string, and Print to stream. When specified, the print option permits printing objects to standard output (the console), user-supplied text buffers (the string), and a user-provided stream in the form of an output callback function.

Note Print to stream is not supported when using Packed Encoding Rules.

Space Optimization Options

The compiler may optimize code for space in several ways. Note that some options are not available for all Encoding Rules.

Indefinite Length Processing Tests

Check Do not generate indefinite length processing code to instruct the compiler to omit indefinite length tests in generated decode functions. These tests result in the generation of a large amount of code. If the compiled application only uses definite length encoding, this option can result in a much smaller code base size.

Note Do not generate indefinite length processing code may only be selected in conjunction with Basic and Distinguished Encoding Rules.

Open Extension Elements

Check Do not generate code to save/restore unknown extensions to prevent the compiler from generating code to read and write elements with extensibility markers. If such generality is not required by the application, check this option to reduce the code size.

Type Constraints

If Do not generate code to check constraints is checked, ASN.1 types with constraints will be treated as types without constraints.

Compact Code

Checking Generate Compact Code instructs the compiler to create more compact code at the expense of some error and constraint checks. It is not recommended to select this option until after the application has been thoroughly tested.

Note Generate Compact Code may not be used when attempting to generate code compatible with previous versions of the compiler.

Compatibility Options

ASN1C may also be instructed to generate code compatible with earlier versions of the compiler. This option may not be used in conjunction with Generate Compact Code.




Generated Type Modifiers

By checking Generate static member variables in choice constructs, the compiler will use static elements instead of pointer variables in generated structures.

Checking Generate short form of type names will cause the compiler to used only the production name and final element name in a type name. Collisions are resolved by appending a unique number to the end of the type name. By default, the compiler will use intermediate names to generate final element names; in deeply nested types, this can result in very long type names.

To resolve type name collisions, check Automatically create unique names for duplicate items.

To cause the compiler to treat all type names as Protocol Data Units (PDUs), check the corresponding box. Default behavior is to treat only unreferenced types within a module as PDUs.

Table Constraint Options

Check Generate code to fully encode/decode items with table constraints to instruct the compiler to generate table constraint processing structures and functions. These add logic for handling types that use message tables for defining what is to be included in a give protocol interaction message. The added logic makes it possible to encode or decode these multi-part messages in a single step.

Note This option is not available for XML Schema or XML Encoding Rules.

Check Enable strict constraint checks on all table constraint items to instruct the compiler to generate code for strict validation of table constraints. Real world messages typically do not follow value field table constraint definitions, so this option should be used with care. It will increase code size and decrease performance.

Event Handler Options

Check Generate code to invoke event handler callback functions to provide support for user-defined event and error handlers. If pure parser functions are desired, also check Generate pure parser.

Output File Options

ASN1C can output resulting code in a variety of ways.

Select Output Code to .c/.h files based on module names to produce files for each ASN.1 Module. If a limit on file length is desired (e.g., to avoid Visual C++ limits), enter the number of lines in the Max lines per file line; the default value is 50,000 lines.

Select Output each generated function to a separate source file to produce a source file for each function. When C is the target language, this will produce a .c file for each generated function. When C++ is the target language, a .cpp file is generated for each control class, type, and C function. This may lead to smaller executable sizes by allowing the linker to only link required object files.

Select Output all code to a single .c/.h file to dump all generated code to a single .c (or .cpp) and .h file.




Makefile Options

ASN1C has the ability to generate makefiles to permit convenient compilation of generated code. Both Windows- and GNU-style makefiles are supported. Static and dynamic linking are also supported. Check Compile all code into a static library to generate a makefile that uses static linking. Check Compile all code into a DLL or shared object file to generate a makefile that uses dynamic linking. To generate a Visual Studio project file, check Generate Visual Studio Project File.

Library Options

By default, ASN1C generates statically-linked, single-threaded libraries. To use shared libraries, select the Generate shared libraries radio button. Check Generate multi-threaded libraries if multi-threading is desired.

Sample Program Generation Options

Check Generate writer sample program to instruct the compiler to produce a program that demonstrates how to populate and encode data structures. Check Generate reader sample program to instruct the compiler to produce a program that demonstrates how to decode and output data structures.

Other Options

To specify other command-line options not usable through the GUI, add them in the provided text line. These options are added at the end of the compilation command line.

Compilation




The compilation command is displayed in the top pane. To execute the compiler, click Compile. Output from the compiler is displayed in the bottom pane. To save the current project, click Save.

If additional changes are required, or to load a new project, hit Back until the necessary window appears.

When finished, click Finished or the X in the upper-right-hand corner of the window.
